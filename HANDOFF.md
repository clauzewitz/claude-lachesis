## 1. 이 앱이 이렇게 생긴 이유 (결정 기록)

### 데이터 소스: 왜 '추정 방식'인가
검토한 선택지와 탈락 사유:
1. **공식 API** — 구독 플랜 사용률을 주는 공개 API가 없음 (Admin API는
   조직용 API 사용량 전용).
2. **비공식 엔드포인트** (`api.anthropic.com/api/oauth/usage`) — 세 항목
   모두 정확히 받을 수 있으나, 구독 OAuth 토큰의 비공식 클라이언트 사용은
   약관 위반 → **계정 정지 위험. 영구 배제.**
3. **상태줄(status line) 캐시 방식** — 공식 기능이라 안전하고 수치도
   정확하지만, 스크립트 설치가 필요하고 Sonnet 전용 항목이 없음.
   사용자가 "설치 없는 방식"을 원해 보류. (초기 버전 코드가 이 방식이었음)
4. **추정 방식 (채택)** — `~/.claude/projects/*.jsonl`의 토큰 기록만으로
   계산. 설치 불필요, 네트워크 0회, 약관 안전. 대신 한도(분모)가
   비공개라 **퍼센트가 공식 /usage와 다름** → UI에 "추정" 배지로 명시.

핵심 제약: 토큰→퍼센트 변환은 원리적으로 부정확함 (절대 한도 비공개,
토큰 종류·모델별 가중치 비공개). 이 한계를 숨기지 않는 것이 디자인 원칙.

### 이름: claude-lachesis
그리스 신화에서 운명의 실 길이를 '재는' 모이라이. 사용자의 기존 상태줄
도구 claude-lynceus(천리안 파수꾼)와 한 쌍: "lynceus는 보는 눈,
lachesis는 재는 손". 앱/타깃명은 Lachesis, 번들 ID는 local.Lachesis.

### 프로젝트 형식 (2026-06-12: XcodeGen 도입)
**`project.yml`(XcodeGen)을 진실원본으로 채택**하고 `.xcodeproj`는 생성물로
전환(git 미추적). 이유: 새 Swift 파일마다 pbxproj 4곳을 손으로 등록하던
통증을 제거 — 이제 파일을 두고 `xcodegen generate`만 돌리면 된다.

연혁(번복 기록):
- (초기) Xcode 16 동기화 그룹 objectVersion 77 → 구버전 Xcode에서 안 열림.
- (이전) 손관리 **전통 형식 objectVersion 56** 으로 교체, Xcode 14/15 호환 확보.
  대신 새 파일 pbxproj 4곳 수동 등록이라는 비용을 떠안음.
- (현재) XcodeGen 도입. **단 XcodeGen 2.45 는 objectVersion 을 77 로
  하드코딩**(옵션으로 못 낮춤)하므로, 위 56 결정은 번복되고 **빌드 최소
  요구가 Xcode 16+ 로 상향**됐다(런타임 대상은 그대로 macOS 13). 수동 등록
  비용을 없애는 대가로 구버전 Xcode 빌드 지원을 포기한 트레이드오프.
- XcodeGen 은 동기화 그룹(synchronized group) 형식은 쓰지 않으므로 소스
  그룹 자체는 전통적 PBXGroup 으로 생성된다(77 = 동기화 그룹이 아니다).
설정 재현 주의는 project.yml 주석과 CLAUDE.md "프로젝트 파일 관련 주의" 참고.

## 3. 해결한 버그 (재발 주의)

1. **링커 오류** `Undefined symbol: fmtRemainingUntil` — 업데이트 시
   일부 파일만 교체해 ContentView(신)와 UsageStore(구)의 짝이 어긋남.
   → 파일 업데이트는 항상 세트로.
2. **EXC_BAD_ACCESS (백그라운드 Task)** — 갱신 겹침으로 공유 static
   ISO8601DateFormatter를 동시 사용해 충돌. 보고된 충돌 줄(elapsed 계산)은
   실제 원인과 다른 위치였음.
   → 수정: refresh 재진입 가드 + 스캔별 지역 포매터 + 미래 시각 필터,
   elapsed 음수 방지, rate.isFinite. **이 가드들 제거 금지 (CLAUDE.md 명시).**
3. **주간 사용량이 /usage와 안 맞음 (2026-06-11 진단)** — 사용자는 "주간 앵커가
   부정확"하다고 의심했으나, 실제 /usage 스크린샷과 대조한 결과:
   - **앵커는 정확**: 계산된 초기화 = 2026-06-15(월) 10:00 = /usage "Resets Jun
     15 at 10am" 일치. 경계 분류·dedup 모두 정상(재시도 중복 계상 0).
   - **진짜 원인 ①**: 주간 합계의 **95.5%가 cache_read**라 전액 합산 시 토큰이
     폭증. → 엔진 합산을 **cache_read×0.1**(과금 정렬)로 변경. 99.9M/434M=23%로
     /usage(23%/1%)와 일치 확인. **1.0배로 되돌리지 말 것.**
   - **진짜 원인 ②**: 자동 한도는 "역대 최대 주간 ≈ 현재 주간"이라 항상
     66~100%로 붙어 실제 23%를 재현 불가. → **수동 한도 모드**로 보정:
     주간 전체 434M / Sonnet 110M / 세션 시드 22M(역대 최대 5h 구간).
     (UserDefaults: limitMode=manual, manualWeeklyLimitM=434 등)
     **※ 이 434/110/22M 값은 2026-06-17 재보정으로 폐기됨 — 아래 §3.5 참고.**
4. **설정 창이 클릭하면 닫힘 (2026-06-11)** — 설정을 MenuBarExtra(.window)의
   임시 패널 위에 .sheet로 올렸더니, 컨트롤을 클릭할 때 패널이 key를 잃고
   팝오버째 닫혔다. (임시 패널은 resignKey 시 자동으로 닫히는 설계)
   → 수정: 설정을 **독립 `Window` 씬**으로 분리(LachesisApp의 Window(id:
   "settings")), ContentView는 `openWindow(id:)` + `NSApp.activate`로 띄움.
   "완료"는 macOS 13에서 dismiss()가 Window를 못 닫을 수 있어
   `NSApp.keyWindow?.close()` 사용. **설정을 다시 .sheet로 되돌리지 말 것.**

5. **수동 한도 재보정 + 부분합 함정 (2026-06-17)** — §3.3에서 정한 434/110/22M이
   이날 /usage와 다시 어긋났다(세 창 모두 앱이 1.7~2.9배 *과소* 표시). /usage
   스크린샷(세션 42% / 주간 33% / Sonnet 2%, 16:01 시점)으로 동기화해 역산·재적용:
   - 적용값: **manualSessionLimitM=15, manualWeeklyLimitM=176, manualSonnetWeeklyLimitM=39.**
     (최종 화면 검증: 세 창 모두 /usage 와 ±1p 이내 — 세션 64↔65, 주간 35↔35, Sonnet 2↔2.)
   - **세션 한도는 한 점으로 못 맞춘다 — "추격 함정"**: 16:01 한 점으로는 13M이 나왔으나,
     이 보정 대화 자체가 cache_read 위주 무거운 작업이라 같은 세션을 16분 뒤 다시 재면
     함의 한도가 13.2M→14.9M→15.2M로 *계속 상승*했다(버스트가 /usage보다 빠르게 누적).
     숫자를 올려 쫓으면 다음 읽기에서 또 올라가는 무한 추격이다. **세션은 5시간짜리
     짧은 창이라 직전 작업 구성에 휘둘리는 가장 노이즈 큰 창**이므로, "정확히 일치"가
     아니라 ±몇 p 추종을 목표로 한다. 함의 한도가 완만해진 15M에서 멈춤(현재 ±1p).
   - **왜 6/11 값이 틀어졌나 — "부분합 함정"**: 6/11의 434M은 주가 *진행 중*이던
     시점의 부분합(99.9M)÷23%로 역산한 값이다. 그 주는 끝나며 152.98M까지 찼다.
     **부분합 스냅샷으로 한도를 역산하면 한도가 과대평가된다.** 보정은 반드시
     "주가 끝난 직후의 /usage %"로 할 것.
   - **드리프트의 원인은 가중치가 아니다**: 6/8~15 주와 6/15~ 주는 모델 구성
     (거의 100% Opus)·cache_read 비중(약 95%)이 거의 동일한데도 함의 한도가
     434M→176M로 ~2.5배 줄었다. cache_read는 raw 토큰의 95%라 가중치(0.1)를
     바꿔도 두 주가 같은 비율로 스케일될 뿐 상대 드리프트는 못 고친다. 즉
     **실제 한도(분모) 자체가 줄어든 것**(Anthropic의 주간 한도 조정으로 추정).
     → `cache_read×0.1`을 건드릴 문제가 아니다. **0.1 가중치 유지(§3.3 ①).**
   - **교훈**: 네트워크 없는 추정이라 한도는 움직이는 표적이다. 수동 한도는
     주기적으로(특히 주 경계 직후) /usage 로 재보정해야 한다. "추정" 배지는 정당.
   - 재현용 스크립트: `/tmp/weekly_diag.py`(주별 가중합·모델 구성),
     `/tmp/session_backcalc.py`(스크린샷 시점 동기화 역산). 엔진 로직을 그대로
     복제하므로 다음 재보정 때 재사용 가능.

6. **퍼센트 표시가 버림이라 1%p씩 낮게 (2026-06-17)** — 재보정 후에도 주간이
   /usage 35%인데 앱은 34%, Sonnet 2%인데 1%로 떴다. 원인은 한도가 아니라 **표시
   레이어가 `Int(percent)`로 버림(truncate)** 하던 것(34.8→34, 1.977→1). 분모는
   옳았고 표시 방식이 틀렸다(특히 Sonnet은 한도 39M이 맞았다는 증거).
   → 수정: 표시 3곳을 `Int(percent.rounded())`(반올림)으로 변경 —
   UsageStore.menuBarText / notifyIfNeeded 본문 / ContentView 카드 퍼센트.
   **임계값 판정(`percent >= 70/90`, barColor·menuBarTint·notify)은 Double 원본
   비교라 건드리지 않았다** — 표시만 반올림하고 판정은 그대로(섞으면 69.6%가 70%로
   보이며 경고 누락 가능). 수정 후 세 창 모두 /usage 와 ±1p 일치.

7. **기본 한도 모드를 auto→manual 로 전환 + 기본값 갱신 (2026-06-17)** — 신규
   사용자(UserDefaults에 저장값이 없는 경우)의 진입 동작을 바꿨다. 그동안 기본은
   auto(역대 최대 자동 추정)였는데, auto는 "역대 최대 ≈ 현재"라 항상 66~100%에
   붙어 /usage 를 재현하지 못한다(§3.3 진짜 원인 ②). 그래서 §3.5에서 역산한
   재보정 한도가 곧바로 분모로 쓰이도록 **기본 `limitMode` 를 manual 로** 바꾸고,
   manual 기본 한도값도 옛 시드(세션 50 / 주간 300 / Sonnet 300M)에서 §3.5
   재보정값으로 갱신했다.
   - 적용 기본값: **limitMode=manual, manualSessionLimitM=17, manualWeeklyLimitM=176,
     manualSonnetWeeklyLimitM=39.** (세션은 §3.5의 15M 대신 17M — 한도는 움직이는
     표적이라 이후 더 최근 값으로 재보정한 결과다.)
   - 변경 지점: 네 키 모두 **3곳씩** 정의돼 있다 — UsageStore(@AppStorage 원본),
     SettingsView 의 저장소 미러(`s*`)와 초안 @State. 세 곳을 항상 같이 맞출 것
     (갈라지면 설정 UI 초안과 실제 계산이 어긋난다).
   - **auto 경로 로직은 그대로다**: `max(관찰최대, 현재)` 추정은 살아 있고, 설정에서
     "자동 추정"으로 전환하면 즉시 다시 동작한다. 바뀐 건 *기본 진입점*뿐이다.
   - 주의: `@AppStorage` 기본값은 저장값이 없을 때만 적용된다. 기존 사용자는 옛
     설정이 그대로 남으므로, 새 기본을 보려면 설정에서 직접 바꾸거나 UserDefaults
